░░░▒▒▓█ SLAG █▓▒▒░░░

Smelt ideas, skim the bugs, forge the product.

Task orchestrator for AI-powered development. Breaks requirements into S-expression ingots and forges them via configurable smith CLIs with strict retry contracts, independent fallback lanes, and proof-based verification.

🧱 ore → 🔥 forge → ✅ done → ❌ fail? → ♻️ re-smelt ↩

Install

curl -sSf https://slag.dev/install.sh | sh

One-liner that downloads the binary and adds to PATH

Other installation methods
# bash version (no build required) curl -fsSL https://slag.dev/slag.sh -o /usr/local/bin/slag && chmod +x /usr/local/bin/slag
# from source cargo install --git https://github.com/sliday/slag

Quick Start

# write your requirements cat > PRD.md << 'EOF' Build a REST API with auth and rate limiting EOF
# forge slag "Build the REST API from PRD.md"

⚠ slag gives the configured smith autonomous shell access. Use in a dedicated directory or container.

Phases

1 SURVEYOR

Deep analysis with plan mode. Reads PRD.md (ore), produces BLUEPRINT.md with architecture, dependency graph, risk assessment, and forging sequence. Self-iterates to resolve any ambiguity.

2 FOUNDER

Casts S-expression ingots from the blueprint. Each ingot has an ID, complexity grade, skill tag, proof command, and work description. Outputs PLAN.md as the crucible.

3 FORGE

Strikes each ingot via the configured smith. Solo ingots run on parallel anvils (up to 3). Selects smith by skill and grade. Retries with slag feedback on failure. Commits on success.

3.5 REVIEW (--worktree)

Master agent quality gate. With --worktree, each ingot forges in an isolated branch. Review runs CI checks (fmt, clippy, test), then AI reviews the diff. Approved branches merge; rejected ones are marked cracked for retry/recovery.

4 ASSAY

Final quality report. Shows forged/cracked counts, temperature bar, and identifies any cracked ingots. Exits 0 on full forge, 1 if any ingot cracked.

Design Decisions

Why S-Expressions?

S-expressions are single-line, grep/sed parseable, require zero dependencies, and survive bash string handling. Every ingot is one line in PLAN.md. The entire orchestrator can manipulate state with sed_i without any JSON/YAML parser. Fields are keyword-prefixed (:id, :status) making them unambiguous to extract with pattern matching.

Why Parallel Anvils?

Independent ingots (:solo t) run concurrently in background subshells, up to MAX_ANVILS=3. This gives 3x throughput for foundation tasks that have no dependencies. Each anvil gets its own smith process. Sequential ingots (:solo nil) run one at a time after parallel work completes.

Why Proof-Based Verification?

Every ingot carries a :proof field containing a shell command. Exit code 0 means pass, anything else means fail. No human review needed. Proofs are concrete: "test -f file", "npm test", "grep -q pattern file". This enables fully autonomous forging with machine-verifiable quality gates.

Why Self-Iteration?

When a surveyor or founder output contains questions, slag detects them and feeds the output back with instructions to resolve autonomously. Up to 3 rounds. This prevents the forge from stalling on ambiguity. The AI is instructed to make expert decisions rather than ask for clarification.

Why Re-Smelt + Reconsider?

When an ingot cracks after exhausting all heats, re-smelting analyzes failure logs, blueprint, and git history to diagnose the root cause. Retry outputs now pass a strict contract: change approach, avoid known failed proof signatures, and provide concrete proof commands. If the primary recovery output is rejected, slag can escalate to an independent fallback smith. If a re-smelted ingot cracks again, a reconsider step triggers — a surveyor-level re-analysis that questions the fundamental approach rather than tweaking details. This gives each ingot two chances to self-correct before permanently cracking: re-smelt (fix) then reconsider (rethink). Only the specific ingot is affected; other anvils continue undisturbed. Analysis now detects sandbox/permission failures (read-only environments where the smith cannot write files) and skips immediately instead of spiraling through futile retries.

Why Metallurgical Metaphor?

Unambiguous vocabulary that maps naturally to the pipeline. Ore (raw input) is surveyed, cast into ingots, heated in a forge, and either becomes forged steel or cracked waste. Every term has exactly one meaning. The temperature gradient (cold → hot → pure) maps to progress from unstarted to complete.

Experiment Loop

Since v1.3.37, slag's forge loop follows an experiment-driven pattern inspired by autonomous research systems:

# the forge heat loop (per ingot)
1. Prepare flux (prompt with blueprint + experiment history)
2. Invoke smith (time-budgeted via :budget)
3. Extract CMD from response
4. Git commit BEFORE verification  (experiment: {id} heat {n})
5. Run CMD + proof verification
6. If pass  → keep commit, record "forged" in ledger
7. If fail  → git revert, record "reverted" in ledger
8. If infra → don't consume heat, retry (up to 6 free)
9. Repeat with structured experiment history in next prompt

Structured Experiment Ledger

Every heat—success and failure—is recorded in TOON (Token-Oriented Object Notation) tabular format in logs/experiments.toon. Field names appear once in the header; rows are compact comma-separated values:

[*]{ts,id,cy,h,smith,status,dur,exit,proof,hash,files,desc}:
  2026-03-28T14:21:03Z,i1,1,1,claude -p ...,forged,19,0,node --check server.js,5f6a2d7,3,Create HTTP server
  2026-03-28T14:21:19Z,i2,1,1,claude -p ...,reverted,35,1,npm test,,2,"CMD failed: TypeError"

On retry, the smith receives compact TOON history instead of verbose prose:

HISTORY(i3)[2]{h,status,dur,err,hash}:
  1,reverted,47,TypeError: foo is not a function,a1b2c3d
  2,reverted,23,"Module not found: bar",e4f5g6h

Git as Memory

Failed experiments stay in git history via git revert (not git reset). The smith can see what was tried by reading the log. Enable with SLAG_GIT_EXPERIMENTS=1 or use --worktree mode (always active there).

Crash Recovery Protocol

Not all failures are equal. Infrastructure failures (smith timeout, rate limit, missing CMD) get free retries without consuming heats:

Failure Type Action Consumes Heat?
Proof fails (exit != 0) Revert commit, log, retry Yes
Smith protocol error (no CMD) Don't commit, retry No (up to 6 free)
Smith timeout / rate limit Failover to next smith in chain No (up to 6 free)
Budget exceeded Timeout, continue next heat Yes

Inspiration

The experiment loop in v1.3.37 was inspired by autoresearch by Andrej Karpathy—a framework that gives an AI agent a training setup and lets it experiment autonomously overnight: modify code, train for 5 minutes, evaluate metrics, keep or discard, repeat ~100 times.

Key ideas borrowed and adapted for general-purpose task orchestration:

  • Git as memory — commit before verify, revert on failure. Failed experiments stay in history for the next attempt to learn from. (via uditgoenka's fork)
  • Structured experiment log — every attempt recorded as JSONL with timing, status, and commit hash. Not just successes—failures are equally valuable data.
  • History-informed iteration — each retry sees what was tried and what failed, not just "try again". The smith gets a complete experiment history.
  • Time-budgeted execution — fixed wall-clock budget per ingot forces creative solutions within constraints, matching autoresearch's 5-minute training cap.
  • Infrastructure vs experiment failures — only real experiments consume retry heats. Protocol errors and timeouts get free retries.

Also influenced by the MLX port which demonstrated hardware-aware experiment optimization on Apple Silicon.

Self-Improvement

Since v1.4.0, anyone can improve slag's codebase and submit a PR:

slag self-improve "Add streaming support to smith output"

Clones from GitHub, forges improvements, measures metrics, creates PR

Predefined shortcuts: quality, tests, performance, tokens. Or pass any freeform text as a commission.

How It Works

1. Check for existing self-improve PRs on GitHub
   [N]ew / [1-3] continue existing / [Q]uit
2. git clone github.com/sliday/slag → /tmp/sandbox
3. Measure baseline (tests, clippy)
4. Forge improvements in sandbox
5. Measure after — compare metrics
6. If improved → gh pr create with metrics table
   If not → discard sandbox

Requirements: gh CLI (authenticated) + any smith CLI. Works from any machine — no need to clone slag manually.

PR Format

Each PR includes a metrics table showing exactly what improved:

## Metrics

| Metric          | Before | After | Delta |
|-----------------|--------|-------|-------|
| Tests pass      | 138    | 143   | +5    |
| Tests fail      | 0      | 0     | 0     |
| Clippy warnings | 2      | 0     | -2    |

slag's first verified self-improvement fixed a clippy warning at config.rs:345 that persisted across 5+ releases. Surveyed its own source → 3 ingots → forged all on heat 1 → outcome PASS → merged.

FAQ

"command not found: slag" after install

The installer adds ~/.slag/bin to your shell profile automatically. Open a new terminal or run:

source ~/.zshrc  # or ~/.bashrc

If that doesn't work, manually add to your profile: export PATH="$HOME/.slag/bin:$PATH"

How do I update slag?

Run slag update to self-update to the latest release. This downloads the new binary from GitHub and replaces the current one.

What's the difference between binary and bash versions?

The Rust binary is faster, has better error handling, and includes self-update. The bash script is a single file with no build step — useful if you can't install Rust or want to inspect/modify the orchestrator directly.

Do I need Claude CLI installed?

Not for the Rust binary. It auto-detects the first compatible smith CLI (claude, then codex, gemini, opencode, Claude-compatible kimi, with native kimi as fallback) unless you set SLAG_SMITH explicitly. If ANTHROPIC_API_KEY is present, auto-detection skips Claude while other supported smiths are available, to avoid accidental API-key billing. You can still force Claude explicitly with --smith claude, --smith claude-plan, or SLAG_SMITH=claude. The legacy bash script still expects Claude CLI.

Ingot Examples

Field Reference

Field Values Meaning
:id "i1", "i2", ... Unique ingot identifier
:status ore | molten | forged | cracked Lifecycle state
:solo t | nil Can run in parallel (t) or must be sequential (nil)
:grade 1-5 Complexity level; grade ≥ 3 uses plan mode
:skill web | api | cli | default Selects smith tools/plugins
:heat 0-N Current retry attempt
:max 5-8+ Max retries before cracking
:smelt 0-2+ Re-smelt/reconsider count (0 = never, 1 = re-smelted, 2 = reconsidered)
:budget seconds (optional) Per-ingot wall-clock time limit for smith invocation
:proof shell command Acceptance test (exit 0 = pass)
:work string Task description for the smith

Progress Display

slag uses emoji indicators to show ingot status at a glance:

[ ✅3 🔥1 🧱5 ] 37%
Emoji Status Meaning
🧱 queued Ingot is ore, waiting to be forged
🔥 forging Ingot is molten, currently being worked
✅ done Ingot is forged, proof passed
❌ failed Ingot cracked after exhausting all heats

The percentage shows overall progress: forged ingots ÷ total ingots.
